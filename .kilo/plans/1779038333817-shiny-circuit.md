## Context & Problem

`src/menu.c` is 1,354 lines of imperative, hand-coded draw/input code. Five pre-game screens exist:

```
GAME_SUBSTAGE_MENU_TITLE (0)  →  DRAW + simple input (CROSS→main)
GAME_SUBSTAGE_MENU_MAIN (1)   →  DRAW + nav + selection (UP/DOWN→cycle, CROSS→call)
GAME_SUBSTAGE_MENU_OPTIONS(2) →  DRAW + nav + row-switch + language modal
GAME_SUBSTAGE_MENU_SETUP(3)   →  DRAW + nav + L/R paging + CROSS commit
GAME_SUBSTAGE_MENU_COLLECTION(4) → DRAW + nav + 2×2 grid
```

**Duplication hotspots across those 5 screens (quantified):**

| Pattern | Occurrences | Lines (imperative total) |
|---------|-------------|--------------------------|
| Backdrop `menu_draw_rect(0,0,SCREEN_W,SCREEN_H,…)` | 5 | 5 |
| Top-strip `menu_draw_rect(0,0,SCREEN_W,6,…)` DARK | 3 | 9 |
| Header-border+text centring | 5 | ~40 |
| `float item_y = … + i * item_h` pattern | 4 | ~20 |
| UP/DOWN wrap-around `if (<0)` / `if (>=N)` branching | 3×2 = 6 | ~24 |
| `MENU_OPTIONS_ITEM_*` label formatting (`snprintf` + value) | 10 rows | ~15 |
| Keyboard hint separator + text | 4 | 12 |
| `graphics_draw_rect(tx,ry,tw,th,…)` for grid cells | collection (4) | ~12 |

**24 individual constant/formatting expressions** repeated across `menu_draw_title`, `menu_draw_main`, `menu_draw_options`, `menu_draw_setup`, `menu_draw_collection`. Any layout or color change requires touching each screen function individually.

---

## Goal

Replace repeated imperative draw code with a **build-time code-generation pipeline** inspired by React's declarative component model:

- **Screens defined as JSON** — layout, colors, children, navigation declared once in data
- **Node.js preprocessor** generates C constants + render functions at build time
- **C runtime library** provides primitive components (`Panel`, `Header`, `ListItem`, `Grid`, `Button`, `Separator`)
- **State binding** (1 line per screen) maps runtime selection index → generated render call
- **Zero JS/runtime on PSP** — only a Node.js script runs during `make`

Result: adding a new screen or changing a color takes writing JSON, not hand-coded C; the render pipeline is 70+ lines shorter per screen; layout changes propagate automatically.

---

## Architecture

```
scripts/uibuilder/                  ← new directory
├── readme.md                       ← usage docs
├── menu_schema.json                ← JSON schema (reference / validator input)
├── generate_menu_c.js              ← Node.js code generator
└── __tests__/

src/ui_gen.c                        ← generated by build step (gitignored or in-tree)

src/ui_runtime.c / src/ui_runtime.h ← new C component library (hand-written, ~200 LOC)
                                      Panel / Header / ListItem / Grid / Button

src/menu_items.json                 ← string tables (labels, values, i18n keys)

menu.c                              ← thin: state binding + e_schema at top
                                      screens call menu_draw_generic(G->json_screen_id)
```

**Data flow:**
1. `menu_screens.json` + `menu_items.json` → input
2. `generate_menu_c.js` → emits `src/ui_gen.c` (C structs + render fns + string literals)
3. PSP build compiles `src/ui_gen.c` along with `src/ui_runtime.c`
4. `menu_draw()` dispatches to `menu_render_title()` / `menu_render_main()` / etc.
5. `menu_update_*()` (hand-written) owns navigation; calls `menu_render_*()` to draw

### Phase 1 — Component library (runtime, hand-written C)

**Goal:** `src/ui_runtime.{c,h}` — the "React runtime" that generated code calls into.

**Exports:**

```c
// src/ui_runtime.h
typedef struct {
    float x, y, w, h;
    uint32_t bg_color;
    uint32_t border_color;
    int border_thickness;
} UIRect;

// Primitives (C functions, no virtual DOM, no heap)
void ui_draw_rect       (UIRect r);
void ui_draw_panel      (float x, float y, float w, float h,
                         uint32_t bg, uint32_t border, int thickness);
void ui_draw_header     (float x, float y, float w, float h,
                         const char *title, uint32_t title_color);
void ui_draw_list_item  (float x, float y, float w, float h,
                         const char *label, const char *value,
                         bool is_selected, uint32_t sel_bg, uint32_t sel_border);
void ui_draw_grid_cell  (float x, float y, float w, float h,
                         const char *label, bool is_selected,
                         uint32_t sel_bg, uint32_t sel_border);
void ui_draw_button     (float x, float y, float w, float h,
                         const char *label, bool is_selected);
void ui_draw_separator  (float y, uint32_t color);
void ui_draw_text_center(const char *text, float cx, float y,
                         float scale, uint32_t color);
```

**New files:** `src/ui_runtime.h`, `src/ui_runtime.c`  
**Updated file:** `src/main.c` — add `#include "ui_runtime.c"` to unity build

**Note on "React-like":** The "React" part is **the build-time generation** (Phase 2), not the C library. The library provides named, composable drawing primitives that generated code calls. No virtual DOM, no JSX, no diffing — just named helpers instead of anonymous `menu_draw_rect` calls scattered throughout hand-written C.

**Rationale for Phase 1 first:** Let us module-test the runtime before the generator writes to it.

#### Build-step wiring

```makefile
# Makefile addition
scripts/uibuilder/generate_menu_c.js: scripts/uibuilder/generate_menu_c.js menu_screens.json menu_items.json
	@echo "  UI-GEN  $<"

src/ui_gen.c: scripts/uibuilder/generate_menu_c.js menu_screens.json menu_items.json
	node scripts/uibuilder/generate_menu_c.js \
		-i menu_screens.json \
		-j menu_items.json \
		-o src/ui_gen.c

src/main.o: src/ui_gen.c src/ui_runtime.c src/ui_runtime.h
```

The `ui_gen.c` target is a dependency of `src/main.o` already covered by the existing wildcard rule, but the explicit `.PHONY`-style rule above ensures `generate_menu_c.js` runs before compilation. `node` must be available in the build environment (Ubuntu runners have it natively; `pspdev/psp-sdk:latest` needs `apt-get install -y nodejs` added to the Dockerfile — a separate ticket).

## What counts as "React"

| React concept | PSP-latro analogue |
|---------------|-------------------|
| Component tree (JSX) | `menu_screens.json` — declarative screen tree |
| `state` + `setState()` drives re-render | Runtime `sel_idx` int → `ui_gen_render_*(sel_idx)` call each frame |
| `props` = immutable config | JSON `colors`, `layout`, `children` baked into generated C `#define`s |
| Virtual DOM diff | **Not present** — PSP has no heap budget for a VDOM |
| JSX transform (build step) | `generate_menu_c.js` — Node.js preprocessor, same role |
| `<Panel /> <Header />` | `ui_draw_panel()` / `ui_draw_header()` in `ui_runtime.c` |
| `<ListItem label="…" value="…" />` | `ui_draw_list_item()` — one call replaces 5–8 draw/text calls |

The "React" analogy is **code structure and build pipeline**, not a JS engine. The generated C is compiled once, VDOM is replaced by compile-time layout.

## Open Questions

| # | Question | Decision Needed |
|---|----------|----------------|
| OQ-1 | **Where does texture draw code live?** `menu_draw_title()` draws logo/ace-card textures via `graphics_set_texture` / `graphics_draw_quad`. Generated code calls named primitives — but texture draws are PSP-specific and not in the runtime library. Option A: hand-written "effect hooks" in `menu.c` called from the generated loop. Option B: add `ui_draw_texture(id, x, y, w, h)` to the runtime. | — |
| OQ-2 | **Do generated functions accept `sel_idx` or a full state struct?** Passing just `sel_idx` is faster (1 int) but requires the generated fn to know which global to read. Passing a `UIScreenState*` is more "React state" but adds indirection. | Present both options; prefer `sel_idx` first, `state struct` later |
| OQ-3 | **`menu_items.json` `snprintf` chains for option labels.** Options screen currently does `snprintf(option_labels[row], …, "Overclock:  %s", val)`. Generator can emit a lookup table `label_fmts[]` + a helper `ui_format_option(row, buf, sizeof buf)`. Option B: embed the `%s` substitution in the generated render loop. | Present both options; align with OQ-2 |
| OQ-4 | **Is `menu_items.json` separate or inlined in `menu_screens.json`?** Separate files give language teams independent access; single file gives one atomic commit. | Present as separate; merge in Phase 3 once stable |
| OQ-5 | **What is the PSP heap budget for generated string tables?** `s_gen_option_labels[10][60]` = 600 bytes. `s_gen_collection_labels[5][32]` = 160 bytes. Total for all screens: ~2 KB — negligible on a 64 MB system. | No action needed |
| OQ-6 | **Should the JSON schema be formal (JSON Schema / AJV) or informal (documentation + silent failure)?** Informal is faster to ship. Formal catches typos at preprocess time. | Start informal; add schema validation in a follow-up |
| OQ-7 | **Node.js version for `generate_menu_c.js`.** `menu_screens.json` is just data — a plain JS script works on Node 12+ (Docker image likely 16–18). No dependency on React or any npm package. | Use stdlib-only; no `package.json` needed |

## Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| JSON screen ≠ pixel-identical C output | Medium | Visual regression bugs | Phase 2 ships generator + visual regression tests before any migration |
| PSP heap hit from generated string tables | Low | Minor memory increase (~2 KB) | Tables are `static const` → go in `.rodata`, not heap; see OQ-5 |
| Build pipeline break if `node` missing in Docker | Medium | CI fails | Add `apt-get install -y nodejs` to Dockerfile; separate tracking ticket |
| State binding out of sync with generator assumptions | Medium | Wrong selection index at runtime | `UI_STATE_MAIN` etc. are `#define`s generated alongside render fns — compiler catches mismatches |
| Over-engineering vs. simpler data-driven C | Low | Unnecessary complexity for ~1354-line file | Phase 3 is opt-in per screen; Phase 1/2 are the only required pieces |
