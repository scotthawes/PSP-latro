# Shader Porting Implementation Plan for PSP

## Overview
This document outlines the plan to port Love2D shader effects from the Balatro reference implementation to the PSP platform. Due to the PSP's fixed-function GPU pipeline, direct shader translation is not possible. Instead, we will implement equivalent effects using CPU preprocessing, multipass rendering, and PSP-specific graphics techniques.

## Shader Analysis

### Shaders in Scope (Recommended for Porting)
These shaders can be reasonably implemented on PSP using CPU preprocessing or fixed-function techniques:

1. **flash.fs** - Simple flash/whiteout effect
   - Complexity: Low
   - PSP Approach: CPU alpha modulation or fixed-function blending
   - Implementation: Modify texture alpha values or use GU_COLOR_8888|GU_TEXTURE_16BIT blending

2. **background.fs** - Animated spinning background with color blending
   - Complexity: Medium
   - PSP Approach: UV transformation + CPU color blending or multipass rendering
   - Implementation: Rotate texture coordinates each frame, apply color modulation

3. **CRT.fs** - Currently transparent (placeholder)
   - Complexity: Very Low
   - PSP Approach: May not need implementation if it's just transparent
   - Implementation: If needed, use constant color output

4. **splash.fs** - Likely simple effect (need to review)
   - Complexity: Low-Medium
   - PSP Approach: To be determined after review

### Shaders Requiring CPU Preprocessing
These shaders are best implemented by preprocessing textures on CPU when cards are created/modified:

1. **foil.fs** - Foil card effect with color shifting
   - Complexity: High (GPU), Medium (CPU)
   - PSP Approach: Preprocess texture when foil applied
   - Implementation: Apply color transformation to texture data before upload

2. **holo.fs** - Holographic/shimmer effect
   - Complexity: High (GPU), Medium (CPU)
   - PSP Approach: Preprocess texture or use animated texture switching
   - Implementation: Shift hues or use multiple texture variants

3. **hologram.fs** - Glitch effect with horizontal offsets and glow
   - Complexity: High
   - PSP Approach: Limited implementation possible with texture animaton
   - Implementation: Simplified version with texture offset animation

4. **negative.fs** / **negative_shine.fs** - Negative/invert effects
   - Complexity: Medium
   - PSP Approach: CPU preprocessing or fixed-function color invert
   - Implementation: Invert RGB values (255-value) during texture processing

5. **played.fs** - Likely played card indicator
   - Complexity: To be determined
   - PSP Approach: Review needed

6. **polychrome.fs** - Multi-color shifting effect
   - Complexity: High
   - PSP Approach: CPU preprocessing with animated color shifts
   - Implementation: Cycle through color palettes over time

7. **voucher.fs** - Voucher card effect
   - Complexity: To be determined
   - PSP Approach: Review needed

### Shaders Requiring Significant Simplification
These shaders use complex algorithms that would be difficult to implement efficiently:

1. **booster.fs** / **debuff.fs** / **dissolve.fs** - Complex dissolve effects with noise fields
   - Complexity: Very High
   - PSP Approach: Significant simplification or alternative approaches
   - Implementation Options:
     - Precomputed dissolve texture animation
     - Simple alpha fade with noise texture
     - CPU-based dissolve mask generation (may be too slow)

2. **flame.fs** - Fire/smoke particle effect
   - Complexity: Very High
   - PSP Approach: Consider replacing with sprite-based animation
   - Implementation: Use animated sprite sheets instead of shader

## Implementation Strategy

### Phase 1: Infrastructure Setup
1. Add shader effect enumeration to graphics.h
2. Create effect processing functions in graphics.c
3. Modify texture loading pipeline to support effect application
4. Add CPU-based texture processing utilities

### Phase 2: Simple Effects (flash, background)
1. Implement flash.fs equivalent:
   - Add global flash time variable
   - Modify rendering to apply alpha flash when active
2. Implement background.fs equivalent:
   - Add background texture with time-based UV rotation
   - Apply color blending via texture combiners or CPU processing

### Phase 3: Card Effect Preprocessing
1. Create card enhancement system:
   - When card gets foil/holo/etc enhancement, preprocess texture
   - Store processed texture variants
   - Use appropriate variant during rendering
2. Implement foil effect preprocessing:
   - Boost blue channel based on color delta
   - Apply contrast adjustments
3. Implement holo effect preprocessing:
   - Apply hue shifts based on time or random offsets

### Phase 4: Dissolve/Transition Effects
1. Implement simplified dissolve system:
   - Use noise textures for dissolve patterns
   - Animate dissolve threshold over time
   - Apply via alpha testing or blending
2. Consider alternatives:
   - Pre-rendered dissolve animation frames
   - Simple radial or linear wipes

### Phase 5: Optimization & Polish
1. Profile CPU impact of texture processing
2. Optimize hot paths (use lookup tables where possible)
3. Consider texture atlasing for effect variants
4. Test on actual PSP hardware

## Technical Details

### CPU Texture Processing Functions to Add
```c
// In graphics.c
void apply_foil_effect(uint8_t* pixels, int width, int height);
void apply_holo_effect(uint8_t* pixels, int width, int height, float time);
void apply_flash_effect(uint8_t* pixels, int width, int height, float intensity);
void apply_dissolve_effect(uint8_t* src, uint8_t* dst, int width, int height, float progress);
```

### Modified Texture Loading Flow
1. Load base texture from file/archive
2. Apply enhancement effects (if any) via CPU processing
3. Upload processed texture to VRAM
4. For animated effects, either:
   - Process each frame (if slow animation)
   - Pre-process multiple variants and animate between them
   - Use shader-like effects via fixed-function where possible

### PSP Fixed-Function Capabilities to Leverage
- Texture coordinate modification (translation, rotation)
- Texture combining (4 stages with modulate, add, etc)
- Alpha testing and blending
- Fog and color coloring (limited)
- Alpha to coverage (if needed)

## Performance Considerations
1. CPU Impact: Texture processing should happen infrequently (on card creation/enhancement, not every frame)
2. Memory: May need multiple texture variants for different enhancement states
3. Fill Rate: Complex multipass effects may impact performance
4. Optimization: Use SIMD where possible, lookup tables for expensive operations

## File Modifications Needed
1. src/graphics.h - Add effect enums and function prototypes
2. src/graphics.c - Implement effect processing functions
3. src/game.c - Integrate with card enhancement system
4. src/draw.c - Modify rendering to use effect variants
5. Possibly src/main.c - For global effect timing variables

## Dependencies
- Existing texture loading infrastructure in graphics.c
- Card data structures in game.c
- Rendering pipeline in draw.c
- Timing/system update mechanisms

## Risks and Mitigations
1. Performance: CPU processing might be too slow
   - Mitigation: Profile early, optimize with lookup tables, limit frequency
   
2. Visual Fidelity: Effects may not match original exactly
   - Mitigation: Focus on recognizable equivalents rather than pixel-perfect matches
   
3. Memory Usage: Storing multiple texture variants
   - Mitigation: Generate variants on-demand and cache, compress where possible
   
4. Development Time: Complex effects may take longer than expected
   - Mitigation: Prioritize effects by visual impact and implementation difficulty
